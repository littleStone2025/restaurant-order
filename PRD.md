## 1. 文档概述

### 1.1 编写目的

本文档旨在描述味道鲜餐厅网页版点单系统的产品需求，明确系统功能边界、交互逻辑、非功能性需求及验收标准，供产品、设计、研发、测试团队使用，作为项目开发的权威依据。

### 1.2 产品背景

随着餐饮行业数字化转型加速，传统纸质菜单及人工点单方式存在效率低、出错率高、人力成本大等痛点。味道鲜餐厅希望通过搭建一套轻量级网页点单系统，让顾客扫码即可完成自助点单，减少等待时间，提升用餐体验，同时降低前厅人力成本。

### 1.3 术语说明

| 术语 / 缩写 | 说明 |
|------------|------|
| PRD | Product Requirements Document，产品需求文档 |
| SKU | Stock Keeping Unit，最小库存管理单元，此处指单道菜品 |
| 购物车 | 用户在当前会话中已选菜品的临时存储集合 |
| 会话 | 用户从扫码打开页面到关闭页面的一次完整使用过程 |
| OTA | Over The Air，指后台实时更新菜品/价格，无需发版 |

### 1.4 参考资料

- 味道鲜餐厅运营手册 v2.3
- 竞品分析报告：美团、口碑、翻台率优化点单系统调研

## 2. 产品概述

### 2.1 产品定位

面向堂食顾客的轻量级 Web 自助点单工具，顾客扫桌贴二维码，无需下载 App，即刻打开网页完成点菜、提交订单，后厨同步接收打印订单。

### 2.2 目标用户

| 用户类型 | 特征描述 | 核心诉求 |
|---------|---------|---------|
| 堂食顾客 | 18-55岁，具备基本智能手机操作能力 | 快速浏览菜单、便捷下单、无等待 |
| 餐厅服务员 | 负责引导扫码、处理特殊诉求 | 减少口头传递，降低出错率 |
| 后厨人员 | 接收并处理厨打订单 | 订单清晰准确，减少反复确认 |
| 餐厅管理员 | 负责菜品维护、数据查看 | 方便更新菜品、查看实时营业数据 |

### 2.3 核心价值主张

- 扫码即用，无需安装，打开速度 ≤ 3 秒
- 菜品图文并茂，支持分类浏览 + 全文搜索
- 购物车实时更新，下单后后厨同步接单
- 支持备注特殊要求（口味/过敏原等）
- 餐厅后台可 OTA 更新菜品信息、沽清标记

### 2.4 系统架构概览

系统采用前后端分离架构：

- **前端**：纯 HTML5/CSS3/JavaScript SPA，无框架依赖，极致轻量
- **后端**：RESTful API（Node.js / Python FastAPI 均可），提供菜品、订单、用户接口
- **数据库**：关系型（MySQL/PostgreSQL）存储菜品、订单；Redis 缓存实时库存
- **部署**：CDN 加速静态资源，API 服务容器化部署（Docker + Nginx）

## 3. 功能需求

### 3.1 功能模块总览

| 模块编号 | 模块名称 | 功能概述 | 优先级 |
|---------|---------|---------|:------:|
| M01 | 首页 / 菜单浏览 | 分类导航、菜品卡片展示 | P0 |
| M02 | 菜品搜索 | 关键词实时过滤 | P0 |
| M03 | 菜品详情 | 图片/描述/标签/规格展示 | P0 |
| M04 | 购物车管理 | 增减/清空/实时计价 | P0 |
| M05 | 下单结算 | 备注/服务费/提交订单 | P0 |
| M06 | 订单状态追踪 | 进度展示/预计出餐时间 | P1 |
| M07 | 后台菜品管理 | 上下架/价格/沽清 | P1 |
| M08 | 数据统计报表 | 销量/翻台率/热门菜 | P2 |

---

### 3.2 M01 首页 / 菜单浏览

#### 3.2.1 功能描述

顾客打开点单页面后，看到完整菜品列表，按分类组织展示，左侧导航栏可快速跳转，滚动时导航自动高亮同步。

#### 3.2.2 功能要点

- 左侧固定分类导航栏，展示分类名称、图标及菜品数量
- 中间菜品卡片网格布局，每列自适应宽度（min 200px）
- 菜品卡片包含：封面图/emoji、菜名、简介（2行截断）、价格、加购按钮
- 角标支持：🔥热销、🆕新品、👍推荐，后台可配置
- Intersection Observer 实现滚动时左侧分类同步高亮
- 点击分类项平滑滚动至对应菜品区域

#### 3.2.3 验收标准

- 菜品列表加载时间 ≤ 1.5s（4G 网络）
- 分类数量 ≥ 5 个时，侧栏可独立滚动
- 菜品图片懒加载，首屏展示占位符

---

### 3.3 M02 菜品搜索

#### 3.3.1 功能描述

顾客在搜索框输入关键词，系统实时过滤菜品，搜索范围覆盖菜品名称、描述、标签。

#### 3.3.2 功能要点

- 搜索框位于菜单区顶部，placeholder 为"搜索菜品名称..."
- 防抖 200ms，减少频繁渲染
- 无结果时展示空状态插图及提示文案
- 搜索结果显示匹配数量
- 清空搜索词恢复全量菜单

---

### 3.4 M03 菜品详情

#### 3.4.1 功能描述

点击菜品卡片（非加购按钮区域）弹出详情浮层，展示菜品完整信息及操作入口。

#### 3.4.2 功能要点

- 弹窗顶部大图（比例 55%）+ 关闭按钮
- 展示：菜名、完整描述、标签组、价格
- "加入购物车"按钮，点击后关闭弹窗并写入购物车
- 后续迭代：支持规格选择（大/小份、辣度）
- 点击弹窗背景层可关闭

---

### 3.5 M04 购物车管理

#### 3.5.1 功能描述

右侧固定购物车面板实时展示当前已选菜品，支持增减数量、清空、备注填写及费用汇总。

#### 3.5.2 功能要点

- 购物车顶部气泡数字实时展示当前菜品总量
- 每条菜品记录展示：emoji、菜名、小计、加减按钮
- 减至 0 时自动从购物车删除
- "清空"按钮需二次确认（Toast 操作，不弹窗）
- 备注文本框：placeholder "如：不要辣、少盐..."
- 费用汇总：菜品合计、服务费（5%）、实付金额
- 购物车为空时"确认下单"按钮置灰不可点

#### 3.5.3 数据持久化

- 购物车数据存储于浏览器 `sessionStorage`，关闭标签页后清空
- 页面刷新时恢复购物车状态

---

### 3.6 M05 下单结算

#### 3.6.1 功能描述

顾客确认菜品、备注后点击"确认下单"，系统提交订单至后端，弹出成功反馈并展示订单摘要及进度状态。

#### 3.6.2 下单流程

| 步骤 | 操作 | 系统响应 |
|:----:|------|---------|
| 1 | 点击"确认下单" | 校验购物车非空，展示加载态 |
| 2 | 系统提交订单 | `POST /api/orders`，携带桌号、菜品列表、备注 |
| 3 | 后端处理 | 生成订单号、写库、推送后厨打印机 |
| 4 | 返回成功响应 | 弹出下单成功弹窗，展示订单摘要 |
| 5 | 清空购物车 | 重置购物车状态，可继续点单 |

#### 3.6.3 订单成功弹窗信息

- 订单编号（WD + 8位时间戳）
- 桌号、菜品列表、备注、实付金额
- 进度条动画：已接单 → 烹饪中 → 即将出餐
- 操作按钮：继续点菜 / 查看订单详情

## 4. 非功能性需求

### 4.1 性能需求

| 指标 | 目标值 | 备注 |
|------|-------|------|
| 首屏加载时间 | ≤ 3s（4G网络） | 静态资源 CDN 加速 |
| 搜索响应时间 | ≤ 200ms（前端过滤） | 本地防抖处理 |
| 下单接口响应 | ≤ 1s（P90） | 接口超时 5s 报错 |
| 页面包体大小 | ≤ 150KB（gzip） | 无重型框架依赖 |
| 并发用户 | ≥ 500 QPS | 单餐厅场景 |

### 4.2 兼容性需求

- **浏览器**：微信内置浏览器（iOS/Android）、Safari iOS ≥ 14、Chrome Android ≥ 90
- **屏幕分辨率**：320px ~ 1920px 自适应响应式布局
- **网络**：支持 4G / Wi-Fi，弱网（2G/3G）下降级展示菊花加载

### 4.3 安全性需求

- **桌号 Token 鉴权**：扫码 URL 携带一次性 Token，防止伪造桌号
- **API 接口限流**：单 IP 60次/分钟，防刷单
- **XSS 防护**：所有用户输入（备注字段）进行 HTML 转义
- **支付安全**：金额计算在后端完成，前端仅展示

### 4.4 可用性需求

- 系统可用性 SLA ≥ 99.9%（月）
- 下单失败需给出明确错误提示（网络异常/菜品沽清/系统繁忙）
- 无障碍：按钮尺寸 ≥ 44×44pt，关键元素 `aria-label` 标注

| 设计要素 | 规范值 | 用途说明 |

| 背景色 | `#F8F7F5` | 页面底色 |
| 卡片背景 | `#FFFFFF` | 菜品卡片、购物车 |
| 主字体 | PingFang SC / 系统无衬线 | 全站文本 |
| 圆角 | 16px（卡片）/ 8px（小组件） | 统一视觉圆润感 |
| 阴影 | `0 2px 16px rgba(0,0,0,0.08)` | 卡片层次感 |

### 5.2 关键交互动效

- **加购飞入动画**：菜品 emoji 飞向购物车角标（600ms 贝塞尔曲线）
- **购物车新增**：slideIn 动画（translateX 从右侧滑入 200ms）
- **弹窗入场**：`scale(0.92)+translateY(20px)` → `scale(1)`（250ms ease）
- **按钮按压**：`:active` scale(0.9)，100ms 反弹
- **Toast 通知**：底部滑出，2.2s 后自动消失

### 5.3 页面布局（桌面端）

| 区域 | 宽度 | 说明 |
|------|------|------|
| 顶部导航 | 100% / 高 64px | 固定吸顶，`z-index: 100` |
| 左侧分类栏 | 220px | 固定定位，可独立滚动 |
| 中间菜单区 | 自适应（1fr） | 主内容区，网格卡片布局 |
| 右侧购物车 | 360px | 固定定位，弹性布局 |

## 7. API 接口规范（概要）

| 接口名称 | Path | Method | 说明 |
|---------|------|:------:|------|
| 获取菜品列表 | `/api/dishes` | GET | 支持 `category_id`、`available=1` 过滤 |
| 获取菜品详情 | `/api/dishes/:id` | GET | 返回完整菜品信息 |
| 获取分类列表 | `/api/categories` | GET | 返回所有分类 + 菜品数 |
| 提交订单 | `/api/orders` | POST | Body: `{table_no, items, note}` |
| 查询订单状态 | `/api/orders/:id` | GET | 返回订单状态、进度 |
| 沽清菜品 | `/api/admin/dishes/:id` | PATCH | Body: `{is_available: false}`（需鉴权） |

**统一响应格式：**

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

> `code` 非 0 时为错误，`msg` 描述错误原因，`data` 为 `null`。

## 9. 整体验收标准

### 9.1 功能验收

- 所有 P0 功能 100% 实现，测试用例通过率 ≥ 95%
- 核心流程（浏览→加购→结算→下单）端对端无阻塞
- 后厨打印机在 30s 内收到订单（局域网环境）

### 9.2 性能验收

- Lighthouse 性能分 ≥ 85（移动端）
- 首屏可见内容渲染（FCP）≤ 2s
- LCP ≤ 2.5s，CLS ≤ 0.1

### 9.3 用户体验验收

- 5位典型用户测试，完成完整点单流程平均用时 ≤ 3 分钟
- 无需培训，首次使用错误率 ≤ 10%
- NPS 满意度问卷得分 ≥ 7/10
